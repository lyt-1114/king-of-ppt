# Content Playbook

Use this when a deck needs richer introductions, more cases, comparisons, evaluation pages, or a technical/report-style structure.

## Complete Deck Spine

For enterprise, AI, platform, product, or research-style decks, build enough context for a reader who was not in the room:

1. Opening position: what the system/product does and why it matters now.
2. Project background: audience, scenario, constraints, and current gap.
3. Pain analysis: 3-4 specific pain points tied to workflow friction or measurable waste.
4. Current solution comparison: old way vs new way, or competitor/tooling comparison.
5. Solution overview: one clear operating model or architecture.
6. Core modules: explain 3-5 modules as a connected system.
7. Mechanism deep dives: 1-3 pages for the most important technical/process mechanisms.
8. Case scenes: 2-4 cases or scenarios with situation, intervention, result, reusable asset.
9. Evaluation: KPI framework, quality controls, risk controls, and evidence.
10. Implementation roadmap: phases, owners, dependencies, and next actions.
11. Future outlook: expansion path, product roadmap, or strategic implications.
12. Summary: what decision or action the audience should take.

Use this as a menu, not a prison. Omit sections the audience does not need.

## Case Page Pattern

Each case page should answer four questions:

- Situation: what workflow or business context was broken?
- Intervention: what changed in data, process, AI, or human review?
- Result: what metric, qualitative outcome, or operational capability improved?
- Reusable asset: what was left behind: rules, state machine, dashboard, prompt library, SOP, model eval set, data map.

Avoid vague case pages that only say "helped efficiency." Name the chain that changed.

## Rich But Not Crowded

More content does not mean more text per slide.

- Expand content by adding pages, not shrinking type.
- Use chapter rhythm: dense evidence pages between calmer divider/summary pages.
- Put one evidence object per slide: table, architecture, case, metric set, timeline, or quote.
- Move detailed lists to notes or appendix if they do not change the decision.
- Use short labels on diagrams; explain nuance in speaker notes or adjacent captions.

## Multi-Style Page System

Use 2-4 compatible page treatments to avoid monotony:

- Hero / thesis page: large statement, minimal support text, strong visual or motif.
- Ink / editorial divider: chapter word, atmospheric texture, one sentence.
- Consulting evidence page: clean grid, metrics, comparison table, precise footnotes.
- Technical architecture page: layered diagram, data flow, modules, inputs/outputs.
- Case scene page: situation/intervention/result columns plus one key metric.
- Evaluation page: radar, scorecard, KPI matrix, or test pipeline.
- Roadmap page: phases, milestones, owners, dependencies.

Keep typography, margins, footer system, and palette consistent across all treatments.

## Chinese Ink-Tech Direction

When a user references an elegant Chinese technology deck, consider this style:

- soft paper background, teal/emerald ink accents, restrained gray text
- large ghosted Chinese character or brush-stroke motif as atmosphere
- bottom wave or mountain line as a subtle recurring footer motif
- fine technical line icons and thin connector lines
- modular diagrams with enough whitespace between units
- stronger section dividers to create rhythm

Use the style as a system, not a decoration layer. If the motif makes content harder to read, simplify it.

## Comparison Pages

Use comparison pages when explaining why a new solution matters:

- Old workflow vs new workflow
- Human-only vs AI-assisted with human review
- Point tool vs operating system
- Demo prototype vs production system
- Cost, speed, quality, risk, traceability dimensions

Limit comparisons to 4-6 rows. Highlight only the decisive differences.

## Evaluation Pages

Evaluation pages should show how the work is judged:

- efficiency: time, throughput, cycle time, manual workload
- quality: error rate, rework, pass rate, consistency
- risk: auditability, review coverage, exception handling
- adoption: active users, handoff completion, process coverage
- learning: feedback loop, sample return, model/version improvement

If numbers are sourced, cite them. If numbers are assumptions, label them as assumptions.
